# 前端设计说明

## 产品视觉定位

前端服务于一个 B 站单平台内容研究产品：用户输入赛道、选题或竞品分析需求，系统返回带真实样本、来源链接、结构化结论与可执行建议的报告。

视觉方向为“成熟动漫编辑部 / 科技内容研究工作台”。它借用编辑网格、分镜框、网点和趋势节点的构图语言，但不使用第三方动漫人物、平台截图、B 站电视图标、品牌粉色或其他受保护素材。

## 设计原则

1. **Evidence 优先。** 来源、结论类型、任务状态和能力边界必须比装饰更醒目。
2. **公共页可以表达，工作台必须克制。** 首页与示例页承担品牌表达；Dashboard、任务和报告页优先保证扫描效率与长内容可读性。
3. **用网格建立秩序。** 标题、编号、分隔线和等宽数据标签形成稳定的信息层级，不依赖大面积渐变、玻璃卡片或巨型圆角。
4. **状态不只靠颜色。** `pending`、`running`、`completed`、`partial`、`failed`、`cancelled` 均有文字标签与对应说明。
5. **功能与 API 不为视觉让路。** 认证、轮询、任务、报告、Evidence、反馈、分享和导出语义保持不变。

## 颜色与字体 Token

颜色集中定义在 `frontend/src/app/globals.css`，并通过 Tailwind 语义名称使用。浅色与深色主题保持同一语义，不在页面内散落独立配色方案。

| 类别 | Token | 用途 |
|---|---|---|
| 页面与容器 | `background`、`surface`、`surface-elevated`、`card`、`popover` | 页面背景、编辑纸张、浮层与卡片层级 |
| 文字 | `foreground`、`muted-foreground` | 正文、说明和辅助元数据 |
| 品牌与强调 | `primary`、`secondary`、`accent` | 靛青主色、深海军蓝区域、少量橙色编辑标记 |
| 语义状态 | `success`、`warning`、`destructive`、`info` | 成功、部分结果、错误和信息提示 |
| 边界与焦点 | `border`、`input`、`ring` | 分隔线、表单边界和键盘焦点 |
| 形状与层级 | `radius-*`、`shadow-editorial`、`shadow-float` | 克制圆角、普通卡片和浮层 |
| 动效 | `motion-fast`、`motion-base`、`ease-editorial` | 交互反馈和入场过渡 |

中文字体使用本地系统回退：`Segoe UI Variable`、`Segoe UI`、`PingFang SC`、`Microsoft YaHei`、`system-ui`。数据编号、Evidence ID 与小型标签使用系统等宽字体，不依赖构建时外网下载。

## 品牌视觉语言

- 保留 `logo-mark.svg`、`logo-wordmark.svg`、`favicon.svg` 与 `Logo.tsx` 的原创播放三角、趋势线和 Evidence 节点组合。
- 首页使用纸张白、编辑网格、有限网点、速度线和青色信息面板；这些元素不会铺满所有页面。
- 报告与工作台使用更安静的平面、细边框和明确分区，确保表格、长 URL、长标题与大量 Evidence 可持续阅读。
- 图标统一使用 Lucide 线性图标，不使用 emoji 充当导航或功能图标。

## 页面层级

- **公共品牌层：** 首页、示例列表、示例详情。强调产品定位、Evidence 方法和 B 站单平台边界。
- **认证层：** 登录、注册。保留品牌语言，但减少装饰并突出表单标签、错误与提交状态。
- **工作台层：** Dashboard、新建分析、任务进度、历史、设置。使用统一 AppShell、桌面侧栏和移动菜单。
- **报告层：** 报告详情、公开分享。正文、结构化结论、Evidence、反馈与附录按阅读顺序组织；公开分享不展示账号所有者的成本和内部轨迹。
- **兼容层：** 旧 `/report/[id]` 继续跳转到正式 `/reports/[id]`，未改变已有路由语义。

## 响应式策略

- 基础内容宽度从 320px 起可用；桌面、1024px 平板和 390px 手机均避免页面级横向滚动。
- 公共 Header 和工作台 AppShell 在移动端切换为可关闭菜单，交互目标保持约 44px。
- 多列信息在较窄视口改为单列或紧凑网格；长 URL、标题、表格和 Evidence 使用断行或局部滚动，避免撑破页面。
- 报告正文限制阅读宽度；桌面附录可吸附在侧栏，手机端回到正文流。

## 动效边界

- 动效只用于入场、菜单、Modal、Toast 和基础交互反馈，主要改变 `opacity` 与 `transform`。
- 动效时长以 160–240ms 为主，不使用持续循环或会引发布局抖动的 hover 缩放。
- `prefers-reduced-motion: reduce` 下关闭平滑滚动并把动画、过渡压缩到近乎即时。

## 可访问性要求

- 提供跳到主要内容的链接、清晰的 `focus-visible` 焦点环和符合视觉顺序的键盘导航。
- 图标按钮提供可访问名称；表单字段使用可见标签，错误靠近对应输入。
- Modal 支持 Escape、焦点进入、Tab 循环、点击遮罩关闭与关闭后焦点恢复。
- Toast 使用 `role="status"` 和礼貌播报；加载、空、错误、部分完成和取消状态均有可读文本。
- 浅色与深色主题均使用高对比正文、边界和交互状态，不以颜色作为唯一信息渠道。
- 打印样式隐藏导航与交互控件，避免卡片、表格和章节标题在 PDF 中被不合理截断。

## 实现与版权边界

本轮复用 Next.js、Tailwind、Radix UI、Lucide、`next-themes` 和既有组件系统，没有新增大型 UI 或动画依赖，也没有修改后端、认证、Cookie/JWT、数据库模型或 API 契约。所有装饰均由 CSS 与项目原创 SVG 构成。
